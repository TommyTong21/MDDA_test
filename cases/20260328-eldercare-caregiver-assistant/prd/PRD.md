# AI 康养护理智能助手（护工端 MVP）PRD

## 1. 背景与问题
你所在的企业是一家全国性康养医疗机构，覆盖全国多个城市，服务老年群体超过 10 万人。机构提供 L1–L6 护理分级服务。当前数据与系统分散（HIS 仅病历、排班系统、纸质记录与 Excel 并行），一线人员记录与协同成本高，突发事件发现与上报存在延迟，交接班信息容易遗漏。

本 PRD 的策略选择：**优先聚焦护工端（移动端/轻应用）**，把“记录—上报—交接”闭环跑通；护士站工作台与家属侧等能力在 Phase 4 以敏捷迭代方式小步补齐。

## 2. 目标与非目标
### 2.1 业务目标（MVP）
- 降低护工日常记录与沟通负担，让“边做边记”成为默认工作流
- 将突发事件上报从“发现后再找人”变为“30 秒内完成上报并自动触达责任人”
- 将交接班从口头经验型传递转为“可追溯的结构化交接补充”

### 2.2 成功指标（建议口径）
- 护工单次记录平均耗时下降 ≥ 40%
- 紧急上报从发起到护士/护士站收到通知 < 30 秒
- 交接班重点事项遗漏率下降 ≥ 30%
- 护工周活跃率 > 70%（试点站点）

### 2.3 非目标（MVP 不做）
- 不做完整护理全流程平台（医嘱、收费、全量病历）
- 不做复杂排班优化与自动派单（仅接入/展示最小任务提醒）
- 不做家属端自助查询与消息服务（仅预留数据结构与通知入口）
- 不做“AI 诊断结论”（仅做结构化提取、摘要、提示，不替代专业判断）

## 3. 用户与角色
### 3.1 第一核心用户：护工
工作特征：高频走动、碎片时间记录、弱网/噪声环境、需要快速上报与交接。

核心诉求：
- 记录不打断节奏（语音/文本/图片都能快速留痕）
- 上报紧急事件必须“快、准、可追溯”
- 交接补充要简单，避免口头遗漏与“重复问答”

### 3.2 协同用户：护士 / 护士站
MVP 中作为“接收与闭环”的角色存在：
- 接收护工上报事件与关键记录
- 对关键字段进行确认/补充

### 3.3 管理者：护士长 / 运营管理
MVP 仅提供最小可见性（可选）：查看事件量、响应耗时与交接完整度。

## 4. 关键场景（MVP）
### 4.1 场景 A：护工现场快速记录（查房/照护过程）
目标：在现场用语音/文本/图片完成记录，系统自动结构化并生成“今日摘要”。

核心流程：
1) 选择老人（床位/姓名搜索）
2) 语音/文本输入（可选拍照）
3) 系统结构化提取（生命体征/饮食/睡眠/排泄/症状/异常/待观察）
4) 护工确认后提交（关键异常需二次确认）

### 4.2 场景 B：紧急上报（30 秒内完成）
目标：跌倒/不适/生命体征异常等事件，快速上报并自动通知责任护士/护士站。

核心流程：
1) 一键进入“紧急上报”
2) 选择事件类型（跌倒/发热/拒食/呼吸异常/生命体征异常/其他）
3) 自动带出老人/房间（或手动选择）
4) 语音/文本补充 + 可选图片
5) 提交后生成事件工单，进入“待处理”，并触达责任人

### 4.3 场景 C：交接补充（结构化 + 可追溯）
目标：交接班时护工用最少输入补充本班重点事项，系统生成交接摘要并可追溯。

核心流程：
1) 选择班次（白/夜）与时间范围
2) 系统自动汇总本班：重点老人、异常事件、未完成事项
3) 护工补充说明（语音/文本）并提交
4) 交接记录存档，护士可查看并补充

### 4.4 场景 D：最小任务提醒（可选，轻量）
目标：将“待办”呈现在护工视角，避免遗漏（不做复杂排班/派单）。

核心流程：
1) 显示今日待办列表（来源：手动创建/护士下发/排班系统最小导入）
2) 护工完成勾选并可附加记录

## 5. 功能需求（MVP 范围）
### 5.1 护工端（P0）
- 老人列表：按床位/姓名搜索，支持标记“重点关注”
- 快速记录：语音/文本/图片输入；结构化字段提取；可编辑确认；提交留痕
- 紧急上报：事件类型选择；自动通知；事件状态流转（待处理/处理中/已处理/归档）
- 交接补充：自动汇总 + 人工补充；交接存档；可查看历史交接
- 最小任务提醒（可选）：今日待办、完成打勾、附加记录

### 5.2 护士/管理端（Phase 4 重点）
- 护士站工作台：多病人总览、异常中心、交接摘要确认
- 家属沟通支撑：统一查询入口、模板化回复、信息可追溯
- IoT / 设备接入：跌倒检测、床垫/手环数据等

## 6. 数据与权限（MVP）
### 6.1 核心对象
- 老人（Resident）
- 护工（Caregiver）
- 照护记录（Care Note）
- 事件（Incident）
- 交接（Handoff）
- 待办任务（Task，可选）

### 6.2 权限原则（MVP）
- 护工仅能访问“自己负责/被授权”的老人（试点期允许按站点范围简化）
- 事件与记录必须留痕（创建者、时间、来源输入）
- 关键异常字段必须可追溯到原始输入（语音转写/文本/图片引用）

## 7. 体验与非功能要求
- 弱网可用：关键操作可本地暂存，联网后自动补传（Phase 4 可增强）
- 性能：首屏 < 2 秒；常见操作 3 步内完成
- 易用：大按钮/大字号/少层级；紧急上报入口明显且防误触
- 安全：全程加密传输；权限隔离；敏感数据不在前端日志输出

## 8. 风险与应对
- 一线抵触：先找 3–5 位种子护工试点；强调减负而非监控；允许纸质并行两周
- 语音识别准确率：护理术语词库；关键字段二次确认；错误可追溯与可纠正
- 数据底子差：MVP 只要求新增数据结构化；历史数据可不回填
- 弱网与设备差异：离线暂存 + 重试；压缩图片；失败可导出待补传列表

## 9. 里程碑（建议）
- Phase 1–3：冻结 PRD/契约与最小接口
- Phase 2 实现：护工端可用闭环（记录/上报/交接）+ 最小验证
- Phase 3：部署与线上验证点（/version、最小 API 走通）
- Phase 4：护士站工作台、家属沟通与更强离线/排班集成
